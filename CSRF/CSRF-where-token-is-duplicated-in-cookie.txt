Lab #6 - CSRF where token is duplicated in cookie
Vulnerability - This lab's email change functionality is vulnerable to CSRF. It uses tokens to try to prevent CSRF attacks, but they aren't fully integrated into the site's session handling system.

Goal - use your exploit server to host an HTML page that uses a CSRF attack to change the viewer's email address.
Credentials: 
- wiener:peter

Analysis:
- Log in with provided account and change email address to see the request and response => see csrf token and csrf cookie are the same
- DELETE CSRF token => Missing parameter 'csrf'
- Change request method to GET => Method not allowed
- Send a request to change email address with an invalid csrf token => invalid csrf token
- Send a request to change email address with an invalid csrf token and an invalid csrf cookie => SUCCESSFULLY
=> can use this vulnerability to attack
=> check wheter exit any cookie setting functionality
- search the blog: "A" => See the response: Set-Cookie: LastSearchTerm=A; Secure; HttpOnly
=> use this to set cookie of victim's browser: csrf=YOUR-KEY; SameSite=None <=> /?search=test%0d%0aSet-Cookie:%20csrf=YOUR-KEY%3b%20SameSite=None
Solution:
- create an HTML form
- note to inject attacker's csrfKey cookie to set csrf cookie of victim's browser
- create a from with a csrf token of attacker's browser and send to victim, and ensure csrf token = csrf cookie
